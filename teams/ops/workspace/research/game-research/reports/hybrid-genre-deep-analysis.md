## 1. 각 장르의 "핵심 재미(Core Fun)" 심층 분해

### 🎲 로그라이크 — "변수와 숙달의 긴장"

#### 표면적 재미 vs 심층 재미

| 구분 | 내용 |
|------|------|
| **표면적 재미** | 다음엔 뭐가 나올까? 두근거림 |
| **심층 재미** | 나의 실력이 쌓이고 있다는 확인 |
| **심리 메커니즘** | 가변 보상 + 역량 성장 = 도박적 쾌감 × 자기효능감 |

#### 로그라이크의 핵심 재미 5가지

**① 절차적 긴장감 (Procedural Tension)**
- 매 런이 고유하기 때문에 "완벽한 공략" 없음
- 플레이어는 항상 새 문제를 풀어야 함
- 같은 빌드도 매번 다른 선택지 → 의사결정이 살아있음

**② 죽음의 교육 기능 (Death as Teacher)**
- 죽음이 처벌이 아닌 학습의 신호
- "아, 이건 이래서 안 되는구나" → 다음 런에서 개선
- permadeath가 가져다주는 "진지함" (각 선택에 무게)

**③ 빌드 발견의 기쁨 (Build Discovery)**
- "이 조합이 이렇게 강하다고?!" 의 발견
- 플레이어가 스스로 찾아낸 시너지 = 소유감
- 개발자도 예상 못한 콤보 → 게임이 살아있는 느낌

**④ 런 서사 (Run Narrative)**
- 각 런은 짧은 드라마: 시작(희망) → 중반(위기) → 결말(승리/패배)
- 30분짜리 영화를 본 느낌 → 반복해도 매번 다른 이야기
- "이번 런은 진짜 아슬아슬했는데..." → 이야기할 거리

**⑤ 메타 진행의 보상 (Meta Progression)**
- 죽어도 뭔가 남는다 → 완전한 실패 없음
- 점점 해금되는 시작 옵션 → 선택의 폭 증가
- "다음엔 이 캐릭터로 해봐야겠다" → 복귀 동기

#### 로그라이크가 요구하는 "플레이어 비용"

```
높음 ←→ 낮음

인지 부하:   [██████████] 높음 (매 선택마다 계산)
시간 투자:   [███████___] 중간 (런 단위, 반복)
감정 투자:   [█████████_] 높음 (죽음의 공포, 승리의 쾌감)
학습 곡선:   [████████__] 높음 (잘하려면 많이 죽어야 함)
```

### 🃏 덱빌딩 — "최적화와 창조의 즐거움"

#### 표면적 재미 vs 심층 재미

| 구분 | 내용 |
|------|------|
| **표면적 재미** | 카드 뽑는 쾌감, 조합 찾기 |
| **심층 재미** | 내가 만든 시스템이 작동하는 것을 보는 것 |
| **심리 메커니즘** | 창조적 소유감 + 시스템 숙달 = 건축가적 만족감 |

#### 덱빌딩의 핵심 재미 5가지

**① 시너지 설계의 쾌감 (Synergy Architecture)**
- "A카드 + B카드 + C카드 = 폭발적 결과" 발견
- 단순 수집이 아닌 "설계"의 감각
- 레고 블록처럼 조합하는 창조적 행위

**② 덱의 진화 서사 (Deck Evolution)**
- 시작: 약하고 엉성한 덱
- 중반: 핵심 콤보 완성
- 후반: 최적화된 "나의 덱"으로 완성
- 덱 자체가 성장 스토리

**③ 의사결정의 깊이 (Decision Depth)**
- "이 강한 카드를 넣을까, 덱 컨셉을 살릴 카드를 넣을까?"
- 정답이 없는 문제를 푸는 즐거움
- 후회와 개선이 반복되는 설계 사이클

**④ 손패의 가능성 공간 (Hand Possibility Space)**
- 5장이 손에 왔을 때 "이번 턴에 뭘 할 수 있을까?" 계산
- 제한된 자원(에너지) 안에서 최대를 뽑는 퍼즐
- 매 턴이 작은 최적화 문제

**⑤ 빌드 정체성 (Build Identity)**
- "나는 독 빌드다", "나는 드로우 엔진 빌드다"
- 빌드가 플레이어의 게임 내 정체성이 됨
- 커뮤니티에서 빌드 공유 → 사회적 경험

#### 덱빌딩이 요구하는 "플레이어 비용"

```
높음 ←→ 낮음

인지 부하:   [█████████_] 높음 (카드 간 상호작용 파악)
시간 투자:   [████████__] 중간-높음 (런 중 결정 누적)
감정 투자:   [███████___] 중간 (실패해도 "덱 문제" → 덜 억울)
학습 곡선:   [███████___] 중간-높음 (카드 메모 + 상호작용 파악)
```

### ⏱️ 방치형 — "기다림과 귀환의 리듬"

#### 표면적 재미 vs 심층 재미

| 구분 | 내용 |
|------|------|
| **표면적 재미** | 숫자가 커지는 것 보기 |
| **심층 재미** | 최소 투자로 최대 성과를 내는 시스템 구축 |
| **심리 메커니즘** | 귀환 보상(Return Reward) + 점진적 과시 = 수동적 성취감 |

#### 방치형의 핵심 재미 5가지

**① 귀환 보상의 쾌감 (Return Reward)**
- 오프라인 동안 쌓인 보상을 한 번에 수집
- "30분 만에 돌아왔더니 이만큼이나 쌓였다!" 의 놀라움
- 현실 시간과 게임이 연결된 느낌

**② 시스템 최적화의 지적 쾌감 (System Optimization)**
- "어떻게 하면 시간당 수익을 최대화할까?" 문제
- 단순한 클릭이 아닌 구조 설계
- 엔지니어링적 사고방식 → 일종의 경영 시뮬

**③ 점진적 과시 (Gradual Showcase)**
- 처음엔 작은 숫자 → 나중엔 1조, 1경 단위
- 숫자 자체가 성취의 증거
- 접두사(K/M/B/T...) 돌파가 하나의 목표

**④ 낮은 압박의 즐거움 (Low-Pressure Fun)**
- 실패해도 잃는 게 없음 (대부분의 방치형)
- 완벽하게 최적화 못해도 진행은 됨
- 스트레스 없이 즐기는 "편안한 게임"

**⑤ 장기 목표 충족 (Long-term Goal Satisfaction)**
- 수주~수개월 단위의 목표 달성
- 일일/주간 체크인의 습관 형성
- "이 게임이 나의 일상에 녹아있다" 느낌

#### 방치형이 요구하는 "플레이어 비용"

```
높음 ←→ 낮음

인지 부하:   [███_______] 낮음 (대기 중엔 생각 불필요)
시간 투자:   [████______] 낮음-중간 (일일 5분이면 충분)
감정 투자:   [██________] 매우 낮음 (실패 없음, 죽음 없음)
학습 곡선:   [███_______] 낮음 (최적화는 선택, 진행은 자동)
```

## 2. 장르 간 충돌과 시너지 구조 분석

### 🔴 근본적 충돌 포인트

#### 충돌 ①: 긴장감 vs 안락함

| 로그라이크 | 방치형 |
|-----------|--------|
| 죽음의 공포로 선택에 무게를 줌 | 실패가 없어 선택이 가벼움 |
| 긴장의 해소가 재미 | 긴장 자체가 없음이 재미 |
| **충돌: 같은 게임에서 양립이 어려움** | |

**해결책:** 레이어 분리 — 방치형 레이어는 긴장 없이, 로그라이크 레이어는 긴장 있게

#### 충돌 ②: 즉각적 집중 vs 분산된 주의

| 덱빌딩/로그라이크 | 방치형 |
|-----------------|--------|
| 지금 이 순간 집중해야 함 | 나중에 확인해도 됨 |
| 실시간 의사결정 | 시간을 두고 최적화 |
| **충돌: 플레이어의 정신적 모드가 다름** | |

**해결책:** 시간 스케일 분리 — 액티브 플레이(집중) ↔ 패시브 성장(방치)을 명확하게 구분

#### 충돌 ③: 런 리셋 vs 영구 성장

| 로그라이크 | 방치형 |
|-----------|--------|
| 죽으면 런 내 성장이 리셋됨 | 모든 것이 누적되고 영원히 남음 |
| 리셋이 장르의 핵심 정체성 | 누적이 장르의 핵심 정체성 |
| **충돌: 같은 게임 내에서 리셋과 누적이 공존해야 함** | |

**해결책:** 이중 진행 구조 — 런 내 성장(리셋) + 메타 성장(영구 누적)으로 분리

---

### 🟢 강력한 시너지 포인트

#### 시너지 ①: 방치 → 덱빌딩 자원 공급

```
방치형이 자원(재화, 카드 조각, 에너지)을 생산
       ↓
덱빌딩에서 그 자원으로 더 나은 덱을 구성
       ↓
더 강한 덱 = 더 빠른 방치형 수익
       ↓ (선순환)
```

**왜 잘 맞나:** 방치형의 "최적화 문제"와 덱빌딩의 "설계 문제"가 동일한 성격

#### 시너지 ②: 로그라이크 런 → 방치형 엔진 업그레이드

```
로그라이크 런을 성공적으로 완료
       ↓
영구 업그레이드 재료 획득
       ↓
방치형 엔진이 더 빠르게 돌아감
       ↓
더 강한 런 시작 가능
       ↓ (선순환)
```

**왜 잘 맞나:** 로그라이크의 "메타 진행"과 방치형의 "영구 누적"이 같은 것

#### 시너지 ③: 덱빌딩 + 방치형 = "자동화 전략"의 창발

```
덱을 구성할 때 "어떻게 하면 자동으로 더 많이 벌까?" 라는
새로운 의사결정 레이어가 생김
       ↓
단순한 "강한 카드" vs "방치 효율 카드" 트레이드오프
       ↓
방치형이 덱빌딩의 새로운 빌드 방향성을 만들어냄
```

**왜 잘 맞나:** 두 장르 모두 "시스템 최적화"라는 핵심이 공유됨

## 3. 비중 모델 3가지 — 핵심 게임플레이 제안

> 아래 3가지 모델은 3장르 중 **하나를 메인(60%)으로 삼고** 나머지 둘이 서포트(각 20%)하는 구조입니다.

---

### 🅰️ 모델 A: **로그라이크 메인** (60:20:20)
#### "전략적 생존 게임에 방치형 엔진을 더하다"

```
로그라이크 60% + 덱빌딩 25% + 방치형 15%
```

#### 핵심 게임플레이 정의

> **"매 런이 독립적인 전략 퍼즐이며, 방치형 성장이 런의 시작 조건을 넓혀준다"**

**플레이 경험 흐름:**

```
[방치 단계] 앱 열면 오프라인 자원 수집 (30초)
      ↓
[준비 단계] 자원으로 이번 런의 시작 덱 강화/선택 (1-2분)
      ↓
[런 단계] 로그라이크 진행 (10-20분, 메인 경험)
  - 층마다 이벤트, 전투, 상점
  - 카드 획득/제거 → 덱빌딩
  - 생사를 가르는 의사결정
      ↓
[결과 단계] 런 완료 → 영구 화폐 획득
      ↓
[강화 단계] 영구 화폐로 방치형 엔진 업그레이드 (30초)
      ↓
[방치 단계] 앱 종료, 자원 자동 생산 (실제 시간 경과)
      ↓ (반복)
```

#### 핵심 재미의 우선순위

1. **1순위:** 런 중 의사결정의 긴장감 (로그라이크)
2. **2순위:** 런 중 덱 진화의 쾌감 (덱빌딩)
3. **3순위:** 오프라인 자원이 다음 런 선택지를 넓혀주는 기대감 (방치형)

#### 방치형의 역할 (서포트)

방치형은 **"런 시작 선택지의 폭"**을 결정하는 메타 레이어로만 작동:
- 오프라인 동안 특정 카드 조각 수집 → 시작 덱에 희귀 카드 포함 가능
- 방치형 자원이 많을수록 "핸디캡 없이" 런을 시작할 수 있음
- 방치형이 핵심 경험을 방해하지 않음 (런 중엔 완전히 액티브)

#### 덱빌딩의 역할 (서포트)

덱빌딩은 **런 중 도구**로만 작동:
- 각 런에서 발견하는 카드로 임기응변적 덱 구성
- 완벽한 덱보다 "이 상황에 맞는 덱"이 중요
- 런이 끝나면 덱은 리셋 (단, 특정 카드 해금은 영구 유지)

#### 게임 디자인 구체 제안

| 요소 | 설계 방향 |
|------|----------|
| **런 길이** | 15-25분 (집중 플레이) |
| **세션 빈도** | 하루 1-3런 |
| **방치 역할** | 런 외 시간에 카드 파편/재화 생산 |
| **덱 리셋** | 런마다 완전 리셋 (시작 덱 풀은 해금으로 확장) |
| **긴장 포인트** | 죽으면 런 보상 감소, 일부는 유지 |
| **방치 상한** | 오프라인 4-8시간치만 축적 (무한 축적 방지) |

#### 대표 레퍼런스 믹스

```
Slay the Spire (코어 경험)
+ Hades (런 서사, 메타 진행)
+ 방치형 리소스 레이어 (loop hero의 자동화 요소)
```

#### 타겟 플레이어

- **메인:** 로그라이크/덱빌딩 팬 (Slay the Spire 1000시간 유저)
- **서브:** 방치형 입문자 (모바일 캐주얼 유저가 진입)
- **세션 패턴:** "하루 한 번 진지하게 + 자기 전 앱 열어 자원 확인"

### 🅱️ 모델 B: **덱빌딩 메인** (60:20:20)
#### "덱이 살아있는 생명체처럼 성장하는 게임"

```
덱빌딩 60% + 방치형 25% + 로그라이크 15%
```

#### 핵심 게임플레이 정의

> **"덱 자체가 장기적으로 성장하는 존재이며, 방치형이 덱에 새로운 카드와 업그레이드를 공급하고, 로그라이크 이벤트가 덱을 시험한다"**

**플레이 경험 흐름:**

```
[방치 단계] 오프라인 동안 덱의 카드들이 "훈련"되어 경험치 획득
      ↓
[덱 확인] 어떤 카드가 성장했는지 확인 (만족감)
      ↓
[덱 편집] 성장한 카드, 신규 카드를 반영해 덱 재조정 (메인 경험 ①)
      ↓
[이벤트 도전] 일일 던전/이벤트에 덱을 데리고 출정 (메인 경험 ②)
  - 이벤트는 로그라이크 구조 (무작위 조우)
  - 5-10분, 덱의 "현재 상태"를 테스트
      ↓
[카드 획득] 이벤트 결과로 새 카드/업그레이드 재료 획득
      ↓
[방치 설정] "이번엔 어떤 카드를 훈련시킬까?" 설정 후 앱 종료
      ↓ (반복)
```

#### 핵심 재미의 우선순위

1. **1순위:** 덱이 천천히 최강으로 성장하는 장기적 쾌감 (덱빌딩)
2. **2순위:** 오프라인 동안 카드가 성장하는 귀환 보상 (방치형)
3. **3순위:** 이벤트에서 예상치 못한 상황 대처 (로그라이크)

#### 방치형의 역할 (강화된 서포트)

방치형은 **"덱 성장의 주 엔진"**:
- 오프라인 시간 동안 카드가 경험치 획득 → 레벨업
- 어떤 카드를 "훈련"시킬지 선택이 전략적 의사결정이 됨
- 방치 설정 자체가 게임의 일부 (5분짜리 의미있는 세팅)
- 돌아왔을 때 성장한 카드 확인 = 방치형의 귀환 보상 감각

#### 로그라이크의 역할 (서포트)

로그라이크는 **"덱을 테스트하는 환경"**:
- 완벽한 덱이 있어도 이벤트는 무작위 → 덱의 유연성 테스트
- 죽어도 덱 자체는 유지 (런 보상만 없어짐)
- 완전한 permadeath 없음 → 덱빌딩 투자가 보호됨

#### 게임 디자인 구체 제안

| 요소 | 설계 방향 |
|------|----------|
| **덱 유지** | 런이 끝나도 덱 유지 (로그라이크와 반대) |
| **카드 성장** | 카드 자체가 레벨 보유, 방치 시간으로 성장 |
| **런 역할** | 덱 테스트 + 새 카드/재료 획득 통로 |
| **방치 역할** | 카드 레벨업, 카드 파편 조합, 희귀 카드 해금 |
| **로그라이크 강도** | 약함 (죽어도 덱 유지, 이벤트만 무작위) |
| **세션 리듬** | 하루 1-2회 확인 + 1회 이벤트 도전 |

#### 특수 메카닉 제안: "카드 훈련 시스템"

```
각 카드는 독립적인 경험치 바 보유
       ↓
방치 설정 시 "훈련 슬롯" (예: 3장) 에 카드 배정
       ↓
오프라인 시간 × 훈련 효율 = 경험치 획득
       ↓
레벨업 시 카드 능력치 향상 또는 신규 효과 해금
       ↓
"레벨 10" 카드는 전설 등급으로 각성 가능
```

이 시스템은 방치형의 귀환 보상을 덱빌딩에 직접 연결하여 **"내 덱이 오프라인에서도 성장하고 있다"** 는 감각을 만들어냄.

#### 대표 레퍼런스 믹스

```
Hearthstone의 카드 컬렉션 시스템
+ Slay the Spire의 카드 시너지
+ Melvor Idle의 스킬 성장 방치 구조
+ 포켓몬의 "키운 캐릭터에 대한 애착"
```

#### 타겟 플레이어

- **메인:** 컬렉션/육성 게임 팬 (포켓몬, 가챠 게임 유저)
- **서브:** 방치형 팬 (성장의 확인에서 재미를 찾는 유저)
- **세션 패턴:** "아침에 카드 확인 + 저녁에 이벤트 한 판 + 자기 전 훈련 설정"

### 🅲 모델 C: **방치형 메인** (60:20:20)
#### "자는 동안도 세계가 돌아가는 전략적 방치"

```
방치형 60% + 덱빌딩 25% + 로그라이크 15%
```

#### 핵심 게임플레이 정의

> **"플레이어는 '감독'이다. 덱(=팀)을 구성하고 전략을 짜면, 팀이 자동으로 던전을 탐험한다. 가끔 위기 상황에서만 직접 개입한다"**

**플레이 경험 흐름:**

```
[복귀 단계] 앱 열기 → 팀(덱)이 자동 탐험한 결과 확인 (1-2분)
  - 어떤 이벤트를 만났는지, 어떤 카드를 얻었는지
  - 성공/실패 로그 확인 (로그라이크 이벤트 결과)
      ↓
[전략 단계] 결과를 보고 덱 재편성 (메인 경험)
  - 약점 발견 → 덱 보강
  - 새 카드 통합 → 시너지 재설계
  - 다음 목표 설정 (어떤 층까지 보낼 것인가)
      ↓
[파견 단계] 팀을 다음 탐험에 파견 설정 (30초)
  - 탐험 방향 설정 (위험한 길 vs 안전한 길)
  - 아이템 사용 여부 결정
      ↓
[방치 단계] 앱 종료 → 팀이 자동으로 탐험 진행
  - 로그라이크 이벤트를 AI가 자동 처리
  - 특정 위기 상황에서만 알림 → 플레이어 개입 요청
      ↓
[개입 단계] (선택적) 알림 받으면 앱 열어 위기 결정
  - "보스 앞인데 HP 낮음, 후퇴할까요?"
  - "희귀 카드 발견, 덱에 추가할까요?"
      ↓ (반복)
```

#### 핵심 재미의 우선순위

1. **1순위:** 팀(덱)이 자동으로 잘 해내는 것을 보는 쾌감 (방치형)
2. **2순위:** 팀의 약점을 분석하고 덱을 개선하는 전략 (덱빌딩)
3. **3순위:** 팀이 예상치 못한 상황을 어떻게 처리했는지 확인 (로그라이크)

#### 덱빌딩의 역할 (강화된 서포트)

덱빌딩은 **"팀 편성의 핵심 전략 레이어"**:
- 카드 = 팀원의 역량
- 덱 구성 = 팀 전략 설계 ("우리 팀은 독 전략이야")
- 플레이어의 주 액션 시간 대부분이 여기에 투자됨
- "이 팀이면 자동으로도 이길 수 있다" 의 자신감 = 만족감

#### 로그라이크의 역할 (배경 엔진)

로그라이크는 **"자동 탐험의 불확실성 생성기"**:
- 팀이 탐험하는 동안 무작위 이벤트 발생
- 플레이어가 직접 결정하지 않아도 AI가 처리
- 가끔 선택을 못하는 "위기 이벤트"만 알림
- 팀이 실패해도 50% 자원은 가지고 귀환 (완전 리셋 없음)

#### 게임 디자인 구체 제안

| 요소 | 설계 방향 |
|------|----------|
| **탐험 AI** | 덱 구성에 따라 AI 행동 패턴 자동 결정 |
| **개입 시스템** | 위기 상황만 푸시 알림 → 플레이어 개입 |
| **실패 처리** | 실패 시 자원 50% 귀환, 덱은 유지 |
| **탐험 시간** | 1시간~8시간 탐험 설정 가능 |
| **덱빌딩 깊이** | 높음 — 플레이어의 주 활동 시간 |
| **로그라이크 강도** | 낮음 — 결과 확인 위주, 직접 플레이 최소 |

#### 특수 메카닉 제안: "자동 전투 AI 학습 시스템"

```
덱의 카드 조합에 따라 AI 전투 스타일이 달라짐
       ↓
예: 방어 카드 다수 → AI가 소극적 방어 전술 채택
   공격 카드 다수 → AI가 공격적 속공 채택
   독 카드 다수  → AI가 장기전 포지셔닝 채택
       ↓
플레이어는 "어떤 AI 스타일로 운영할지"를 덱으로 결정
       ↓
탐험 결과 로그에서 AI가 어떤 카드를 어떻게 썼는지 확인 가능
       ↓
개선점 발견 → 덱 수정 → 재파견
```

이 시스템은 방치형의 "자동화 경험"에 덱빌딩의 "설계 재미"를 완벽하게 결합.

#### 대표 레퍼런스 믹스

```
Loop Hero의 자동 전투 + 플레이어 파견 구조
+ Slay the Spire의 카드 시너지 깊이
+ Egg Inc.의 귀환 보상 리듬
+ Adventure Capitalist의 투자/수익 최적화 감각
```

#### 타겟 플레이어

- **메인:** 바쁜 직장인/부모 (시간 없지만 전략 즐기는 유저)
- **서브:** 방치형 고수 (최적화에 진심인 유저)
- **세션 패턴:** "아침 출근 전 5분 덱 조정 + 저녁 결과 확인 + 주말 덱 대대적 개편"

## 7. 결론 및 권고사항

### 핵심 인사이트 3가지

**① 충돌은 분리로 해결한다**
세 장르의 충돌은 같은 시간대에 같은 장르를 요구할 때 발생합니다.
시간 스케일 분리 (런 중 ↔ 런 밖), 레이어 분리 (액티브 ↔ 패시브)로
세 장르가 서로 방해하지 않고 보완할 수 있습니다.

**② 하나의 장르가 "정체성"이어야 한다**
"세 장르 모두 잘하는 게임"보다 "로그라이크 게임인데 방치와 덱빌딩도 있다"처럼
명확한 정체성이 있어야 마케팅도, 유저 기대치 관리도 쉬워집니다.

**③ 방치형은 항상 "접착제" 역할을 한다**
어떤 모델에서든 방치형은 독립적인 핵심이라기보다,
게임의 시간 구조를 만들고 유저 복귀 동기를 제공하는
인프라 레이어로 작동할 때 가장 효과적입니다.
