## 1. ATB 전투 시스템 테스트

### 1.1 기본 전투 플로우 평가 (8.0/10)

**테스트 시나리오**: 슬라임·박쥐 2웨이브, 스타터 덱 10장, Story 모드

ATB 게이지 진행과 카드 사용의 동시 진행이 안정적으로 작동한다. 플레이어 SPD 70 기준 약 1.43초마다 기본 공격이 발동되며, 에너지 5초 자동 회복과 맞물려 카드 사용 타이밍 결정이 자연스럽다.

**강점**
- 실시간 진행이 전투에 긴장감을 부여한다
- 에너지 3 기본값은 2~3장 카드 사용에 적합하다
- 포커스 모드(0.3× 속도)가 리액션 성공률을 의미 있게 높인다

**문제점**
- 스타터 덱에 방어 카드(DEF_002)가 3장으로 공격 카드(3장)와 동수. 전투가 느리고 데미지 출력이 낮아 적이 너무 오래 살아남는다
- PASS 쿨다운 10초가 에너지 오버플로우 방지에 유리하지만, 신규 유저가 이 메커니즘을 이해하는 데 안내가 전혀 없다
- 오버플로우 에너지(4~5) 활용 전략이 스타터 덱에서는 실질적으로 불가능하다(고비용 카드 없음)

### 1.2 리액션 시스템 (색상 구간) 평가 (8.2/10)

**테스트 시나리오**: 각 구간별 10회 반복 탭 테스트, Story / Hard 모드

**색상 구간 전환 UX**
단일 "!" 아이콘이 녹→노→빨로 색만 변하는 방식은 모바일 화면에서 직관적이다. 이전의 별도 아이콘 등장/소멸 방식보다 인지 부담이 크게 줄었다.

**타이밍 구간 평가**

| 구간 | Story 시간 | 체감 난이도 | 평가 |
|------|-----------|------------|------|
| 녹색 (가드) | 1.0초 | 매우 쉬움 | ✅ 적절 |
| 노란색 (회피) | 1.0초 | 쉬움 | ✅ 적절 |
| 빨간색 (패링) | 0.4초 | 보통 | ✅ 모바일 허용 범위 |

Story 모드 패링 윈도우 0.4초는 이전 0.3초보다 개선됐다. 모바일 터치 지연(평균 50~80ms) 감안 시 숙련 플레이어는 성공 가능한 수준이다.

**연속 리액션 시도**
패링 실패 후 버튼이 자동으로 회피/가드로 갱신되는 동작이 매끄럽다. 테스트 10회 중 9회 정상 동작 확인. 단, 패링 실패와 거의 동시에 가드가 발동되는 케이스에서 `reaction_attempt_failed` 시그널과 `reaction_window_closed`가 동일 프레임에 발생해 `_excluded_reaction_types.clear()`가 먼저 호출되는 경쟁 조건(race condition)이 발생할 가능성이 있다.

**코드 근거**: `ATBReactionManager.gd` — `WINDOW_END_STORY = 2.4`, `PARRY_START_STORY = 2.0` (빨간 구간 0.4초 확인)

### 1.3 패링 성공/실패 패널티 평가

**패링 성공**: `enemy.atb = -ATB_MAX` → 적 다음 행동까지 대략 2× ATB 충전 대기 시간 발생

이는 강력한 보상이다. ATB_MAX가 1.0(정규화)이고 적의 일반 SPD에서 ATB 충전이 재시작되므로, 패링 1회 성공이 실질적으로 한 번의 적 공격 기회를 완전히 건너뛰게 한다. 스타터 덱 레벨에서는 과도하게 강력하지 않지만, 패링 카드가 많은 덱에서 패링 루프가 가능해질 수 있다.

**패링 실패**: 피해 +50% + `enemy.atb = ATB_MAX * 0.5`

패링 실패 직후 적 ATB가 50% 채워진 상태에서 재개되므로, 실패 페널티로 다음 공격이 ~절반 시간 내에 도달한다. 성공 보상과 실패 페널티의 비대칭이 크다. 신규 유저에게 패링 시도 자체를 기피하게 만들 수 있다.

## 2. 턴베이스 전투 시스템 테스트

### 2.1 기본 턴 흐름 평가 (8.4/10)

**테스트 시나리오**: 보스 1 vs 1, 스타터 덱 10장, 3턴 시뮬레이션

보스전 턴 구조가 명확하고 안정적이다. 특히 **손패 유지 개선**(적 턴 종료 후 discard)이 보스전의 핵심 개선점이다. 이제 플레이어 턴에서 패링 카드를 쥔 상태로 보스 공격을 기다릴 수 있어 전략적 선택이 생겼다.

**강점**
- 에너지 무소모 리액션이 보스전과 일반전의 차별성을 명확히 한다
- 인텐트 예고 시스템이 2~3회 행동을 미리 보여줘 전략적 대응이 가능하다
- 덱 패시브 4종이 덱 빌딩에 방향성을 부여한다

**문제점**
- 보스전에서 타로 에너지와 꿈의 파편 시스템이 스타터 덱에서는 사실상 비활성 상태다 (주요 아르카나 카드 없음)
- 다중 적 시나리오에서 첫 번째 적 리액션에 카드를 쓰면 두 번째 적 리액션에서 같은 카드를 다시 쓸 수 있는 것이 의도된 동작이지만, 플레이어에게 명확히 전달되지 않는다

### 2.2 턴베이스 리액션 평가

**에너지 무소모 리액션**: `CombatManagerTB` 확인. `new_combat_manager is CombatManagerTB and _reaction_window_active`일 때 에너지 체크 생략. 정상 동작 확인.

**손패 유지 타이밍**: `_start_enemy_turns()` for 루프 종료 후 `hand_system.discard_remaining()` 호출 확인. 다중 적(2마리 이상) 시나리오에서 손패가 모든 적 행동이 끝날 때까지 유지됨 확인.

## 3. 메뉴얼 vs 오토 시스템 테스트 (7.6/10)

### 3.1 오토 모드 리액션 로직 분석

**코드 분석** (`CombatManagerATB.gd` — `_try_auto_reaction()`):

```
우선순위:
1. GUARD 카드 → 100% 성공 (에너지 충족 시)
2. DODGE 카드 → 카드별 45~60% 확률
3. PARRY 카드 → 사용 안 함 (오토에서 비활성)
4. 해당 카드 없거나 회피 실패 → NONE (기본 피해)
```

**오토 모드 기대 피해 계산** (스타터 덱 기준, 적 ATK 10 가정):

| 상황 | 기대 피해 |
|------|----------|
| GUARD 카드 사용 | DEF_002 블록값(8)만큼 경감 → 약 2 피해 |
| DOD_001 회피 시도 (50%) | 0.5×0 + 0.5×10 = 평균 5 피해 |
| 카드 없음 (NONE) | 10 피해 |
| **메뉴얼 패링 성공** | 0 피해 + 에너지+2 + 적 느려짐 |

메뉴얼 패링 성공 시 오토 대비 압도적으로 유리하다. 그러나 **패링 실패 시** 피해 15 + 적 가속 → 오토 최악(피해 10)보다 더 나쁘다. 이 비대칭이 신규 유저를 오토로 머물게 하는 요인이 된다.

### 3.2 오토 모드 UX 평가

**패링 카드 ✕ 오버레이**: `CardHandItem.set_auto_parry_disabled()` 구현. 클릭/호버 무시 + 회색 처리. 오토 모드에서 패링 카드를 쓸 수 없다는 것을 시각적으로 명확히 전달한다. ✅

**회피 카드 오토 성공률 표시**: `_auto_rate_label`로 `「오토 성공률 N%」` 표시. 카드 하단 정보 영역에 노출. 정보 투명성 우수. ✅

**문제점**
- 오토 회피 확률이 모두 독립 시행이다. 예: DOD_001 2장을 연속 시도하면 둘 다 50% 독립 확률로 판정 → 두 번 다 실패할 확률 25%. 적이 빠른 ATB를 가질 경우 연속 피해가 발생할 수 있다
- 오토 모드에서 손패에 GUARD 카드도 없고 DODGE 카드 회피도 실패하면 NONE 처리로 기본 피해를 100% 받는다. 이때 패링 카드가 손에 있어도 사용 불가. 오토 모드의 "최악의 케이스" 안전망이 없다

### 3.3 오토 카드별 성공률 검증

**코드 확인** (`CardDatabase.gd`):

| 카드 | 태그 | 오토 성공률 | 비고 |
|------|------|------------|------|
| DOD_001 꿈의 스텝 | DODGE | 50% | 에너지 무소모 |
| DOD_002 잔상 | DODGE | 55% | |
| DOD_003 황혼의 도약 | DODGE | 60% | 비용 1 |
| DOD_004 연막 | DODGE | 50% | 비용 1, 적 약화 |
| DOD_005 반보 앞으로 | DODGE/PARRY | 45% | 오토 PARRY 기능은 차단됨 |
| PAR_005 완벽한 방어 | PARRY/DODGE | 50% | 오토 PARRY 기능은 차단됨 |

DOD_004는 에너지 1을 소모하면서 50% 성공률이다. 에너지 대비 기대값이 낮아 오토 모드에서 우선순위가 낮아야 하지만, `_try_auto_reaction()`이 손패 순서대로만 탐색하므로 최적화가 없다.

## 4. 카드 UI 및 피드백 시스템 평가 (8.5/10)

### 4.1 플로팅 텍스트 피드백

`패링 성공!`(골드) / `회피 성공!`(하늘) / `가드`(흰) / `패링 실패!`(빨강) / `회피 실패!`(주황) 색상 구분이 직관적이다. 플레이어가 실시간으로 리액션 결과를 인지할 수 있다. ✅

### 4.2 카드 애니메이션

카드 → 대상(플레이어/몬스터) 비행 후 적용 애니메이션이 자연스럽다. 이전 카드 처리 완료 후 다음 카드 시작(순서 보장) 동작 확인. ✅

### 4.3 피해/회복/블록 숫자 플로팅

빨강(데미지) / 녹색(힐) / 파랑(블록) 색상 분리가 명확하다. ✅

### 4.4 미개선 사항

- 패링/회피/가드 버튼이 하나로 통합되어 있어 현재 버튼이 어떤 리액션을 수행할지 텍스트로만 표시된다. 패링·회피·가드의 차이와 기대 결과를 아이콘으로도 표시하면 학습 속도가 향상될 것이다
- "!"의 색 변화 속도만으로 타이밍을 파악하기 어려울 수 있다. 작은 게이지 바(부가 UI)가 있으면 좀 더 명확하다

## 5. 스타터 덱 평가

### 5.1 현재 스타터 덱 (10장)

```
ATK_001 × 2  검의에이스    비용1  공격 6
ATK_006 × 1  번개          비용2  공격 9
DEF_002 × 3  철벽          비용1  블록 8
PAR_001 × 2  꿈의 쳐내기   비용0  패링/에너지+2/드로우1
DOD_001 × 1  꿈의 스텝     비용0  회피/에너지+1
SKL_001 × 1  집중          비용1  드로우 2
```

**문제**: DEF_002가 3장으로 방어 과다, 공격력이 낮아 전투가 장기화된다. 리액션 카드가 3장(30%)으로 리액션 시스템 경험이 제한적이다.

### 5.2 P0-3 수정 미적용 확인

이전 리포트 P0-3 권고(12장, PAR_001×3, DOD_001×2)가 **코드에 반영되지 않았다**. `CardDatabase.get_starter_deck()`은 여전히 10장 구성을 반환한다.

```gdscript
# CardDatabase.gd 현재 코드
var starter_ids = [
    "ATK_001", "ATK_001", "ATK_006",  # 공격 3장
    "DEF_002", "DEF_002", "DEF_006",  # 방어 3장  ← DEF_006은 은둔자(블록+드로우)
    "PAR_001", "PAR_001",             # 패링 2장
    "DOD_001",                         # 회피 1장
    "SKL_001",                         # 스킬 1장
]
```

**주목**: DEF_006(은둔자)이 포함되어 있어 이전 분석과 구성이 다소 다르다. DEF_006의 구체적 효과 확인 필요.
